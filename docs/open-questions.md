# 議論しなきゃいけないこと (Open Questions)

`concept.md`で示した方向性のうち、まだ詰まっていない論点を集めたもの。重要度が高そうな順。

---

## 1. タスクの粒度と分解は誰がやるか

「意図 → タスク」の変換は誰の責務か。

- 人間が意図だけ書いてエージェントがタスク分解する場合、**分解の妥当性自体が新しいレビュー対象**になる
- タスク粒度が大きすぎると検証経路が混ざりすぎ、細かすぎるとオーバーヘッドが爆発する
- タスク分解そのものを承認フローに乗せるか、エージェントに任せるかは設計の分岐点

---

## 2. 並列エージェントの調停モデル

ADR 0009〜0018 で主要部分は確定済み。残る論点:

- **意味レベルのコンフリクト検出**: ADR 0017 の通り「テキストコンフリクトしないが意味的に壊れる」ケースは現状検証経路の品質に依存。シンボル単位の追加検出レイヤーを将来持つかは未決定
- **同一Intent内のTask間順序**: ADR 0018 で対象外と明記。当面はエージェント側で順序付け
- **依存待ちTaskの細かい運用**: ADR 0018 で「claim可だが実装進められない」と決めたが、具体的なAPI挙動は別途設計

## 3. 「意図」の永続性と陳腐化

ADR 0019〜0024 で主要部分は確定済み。残る論点はなし(必要が出れば再オープン)。

## 4. プライバシー・セキュリティモデル

ADR 0025〜0028 で初期版は確定済み。残る論点:

- **token のライフサイクル**(発行・失効・ローテーション)の具体仕様
- **Session 管理API**(一覧・切断)の仕様
- **capability/scope を token に持たせる将来拡張**(初期は不要、リモート連携時に必要になる見込み)
- 削除要件(GDPR等)とappend-only監査ログの整合(将来検討)

---

## 5. オフライン性・分散性

ADR 0029 で MVP は単一開発者スコープと決定。将来拡張時の論点:

- マルチ開発者対応の方式選択 (hub-and-spoke / git的分散 / federation)
- Intent/Task/claim の同期プロトコル
- claim 整合性の保証方式(オフライン中の claim をどう扱うか)
- 監査ログ(ADR 0027)の分散マージ

これらは MVP 検証後に再評価。

---

## 6. エコシステム・移行戦略

ADR 0030〜0032 で MVP 段階の方針は確定。MVP 後の論点:

- 既存 GitHub リポジトリからの移行パス(過去のコミットに対する Intent / Task の遡及付与)
- 最初に統合すべき外部ツールの優先順位(ユーザーフィードバック次第)
- ライセンス選定の確定(MIT / Apache 2.0 / 他)
- コミュニティガバナンス(コントリビューション体制、RFC プロセス等)

---

## 7. エージェントのコスト・レイテンシ

ADR 0033 で確認: hoy 本体は LLM 推論コストを内部負担しない設計。原版の主要な懸念は他ADRで解決済み。

残る論点(MVP 実装段階で決める):

- daemon の性能目標値(API レイテンシ、メモリ、スループット)
- 検証経路の並列実行ポリシー
- 大規模 Intent/Task 集合での クエリ性能

---

## 8. 失敗モードの設計

ADR 0034〜0035 で主要部分は確定。MVP実装段階で詳細化が必要な点:

- reconciliation の具体的な意味論(乖離時に Git/daemon のどちらを正とするか)
- バックアップ・リストアのフォーマット
- 統計APIの具体仕様(Principal別レート、claim履歴等)
