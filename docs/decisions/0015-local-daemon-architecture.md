# 0015. hoyはローカル常駐daemonとして動作する

- **Status**: Accepted
- **Related**: open-questions #2, #5

## Context

hoy のランタイムをどう構築するか。リモート中央サーバ型(GitHub的)、ローカル常駐型(git的)、ハイブリッドなどの選択肢があった。

## Decision

hoy はローカルマシン上で動く **常駐 daemon プロセス**として実装する。daemon は以下を担う:

- 状態管理(Intent / Task / 検証経路 / claim)
- コンフリクト検出と統合処理
- claim 管理(ハートビート、強制release)
- MCP サーバ
- イベント発行と hook 起動

GitHub に対する git のように、リモートサーバは初期スコープ外。

## Rationale

- 「サーバ側実行 vs クライアント側実行」の議論が消え、設計が単純化される
- daemon と hook が同じマシンにいる前提で、エージェント連携が綺麗に書ける
- セキュリティ・スケーラビリティの懸念(任意スクリプトをリモート実行する設計)が初期スコープから消える
- 単一マシンの開発体験を最初に固められる
- リモート同期(複数開発者・"origin" 相当)は将来拡張として open-questions #5 で扱う

## Consequences

- 現時点ではマルチマシン協調は未対応
- daemon はバックグラウンドで常時動く想定。停止時の挙動・自動起動は別途設計
- リモートサーバが必要になった場合は、daemon 同士の同期プロトコルとして設計する(初期 daemon を捨てて再設計しない)
- 開発者ごとに独立した状態を持つことが前提。「皆の認識する main」が分裂するリスクは将来の同期設計で解決する
