# 0014. Task完了時に即時統合する

- **Status**: Accepted
- **Related**: open-questions #2

## Context

Task が完了(必須の検証経路すべて passed)になったとき、変更セットを「公式の状態」(main相当)にいつ・どう取り込むかを決める必要があった。

選択肢:
- (a) 即時統合: 完了 → 自動でmainに取り込み
- (b) 完了キュー: 別機構で統合タイミングを制御
- (c) Intent単位統合: 紐づくIntent完了時にまとめて統合

## Decision

Task が完了した瞬間に **即時統合**する。検証経路がすべて passed なら自動で main 相当に反映される。

## Rationale

- nextgit の初期プロダクト価値として「ガードレール通過 → 即マージ → 人間は細かくレビューしない」を打ち出す
- 「PR を人間が読んで承認する」儀式の置き換えを明確に表現する設計
- 完了から反映までのラグを最小化することで、後続の Task が古い main を前提に作業する事故を減らす
- (b)(c) は柔軟だが「完了したのに反映されない」中間状態を生み、初期の単純さを損なう

## Consequences

- 検証経路の設計品質が直接「壊れた変更が main に入るリスク」を左右する → ガードレールの設計が極めて重要
- 同時に複数 Task が完了した場合の順序付け・コンフリクトでの待ちが発生 → 別途設計が必要
- 統合に失敗した(コンフリクト等)場合の差し戻しフローも別途設計
- Intent 単位での整合性保証は弱まる(中途半端に Task だけ統合された状態が main に存在しうる)
- 将来的に「重要変更は Intent 単位で統合」のような上位ポリシーを足す余地は残す
