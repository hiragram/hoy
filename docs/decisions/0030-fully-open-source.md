# 0030. hoyはフルOSSとして開発する

- **Status**: Accepted
- **Related**: open-questions #6

## Context

hoy のライセンス・事業戦略を決める必要があった。プロトコルが普及するかは設計の前提に直結する。

選択肢:
- (a) フルOSS
- (b) プロトコルOSS + ホスティング商用
- (c) コア商用
- (d) Open Core

## Decision

**フルOSS** として開発する。daemon、CLI、プロトコル仕様すべて公開。

## Rationale

- プロトコルが普及しなければ「GitHubの代替」を名乗る価値がない
- 「プロトコルが正しければホスティングサービスは後から何でも作れる」(concept.md §5.2)という設計思想と整合
- ローカル常駐 daemon (ADR 0015) という配布形態は OSS と相性が良い
- 商用化を最初の目的に据えると、初期の設計判断が事業要件に引きずられがち
- 将来「ホスティングを商用で提供する」選択肢は OSS と両立する。MVP段階で決める必要はない

## Consequences

- ライセンス選定は別途必要(MIT / Apache 2.0 / BSL 等)。商用利用の制限は当面入れない方針
- プロトコル仕様は公開ドキュメントとして整備する
- 拡張ポイント(hook、MCP インタフェース、検証経路の種別等)を明示的に設計する
- コミュニティからの貢献を受け入れるためのガバナンス・コントリビューション体制が必要(MVP後)
- 「事業として何で稼ぐか」は本ADRの対象外。将来的にホスティング・サポート・OSS互換の商用サービス等の選択肢は残る
