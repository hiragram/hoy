# 0028. 機密管理機構は内蔵しない、OSファイル権限+外部secret manager前提

- **Status**: Accepted
- **Related**: open-questions #4

## Context

センシティブな内容(人事意思決定、セキュリティ脆弱性、認証情報など)がIntent本文に含まれる可能性に対し、nextgitが暗号化・アクセス制御を内蔵するかを決める必要があった。

選択肢:
- (a) 暗号化なし(OSファイル権限のみ)
- (b) Intent単位で暗号化フラグ
- (c) 機密管理は外部システムに任せる

## Decision

**(a) + (c) の組み合わせ**:

- nextgit daemon は機密管理機構を**内蔵しない**
- ローカル daemon 前提のため、保管中のデータは **OSファイル権限**で守る
- 真に機密性が必要な情報(認証情報・トークン等)は**外部のsecret manager**(1Password, vault, OSキーチェーン等)に置き、Intent からは参照のみ書く

## Rationale

- ADR 0015 のローカル daemon 前提では、OS の権限モデルで一定のラインは守れる
- 暗号化機構を内蔵すると鍵管理という新しい責務が生まれ、初期スコープを大きく超える
- 暗号化は「内蔵しないと不可能」ではなく、リポジトリレベルで `git-crypt` 等の既存ツールが使える
- 機密の扱いは ecosystem 全体の問題であり、nextgit が独自モデルを持つより既存の secret manager に統合する方が現実的

## Consequences

- 開発者は機密情報を Intent / Task 本文に直書きしないという運用規範を持つ必要がある
- daemon の保管ディレクトリは適切な権限(例: `0700`)で作成される
- 将来的にリモート連携(open-questions #5)が現実化した場合、本ADRは再検討が必要
- 「Intent内の文字列が機密に見える」を検知する linter のような hook はプロジェクト側で実装可能
- 監査ログ(ADR 0027)にも機密本文が直書きされた場合は同じリスクを負う点に注意
