# 0013. Task変更セットは内部的にGitを再利用する

- **Status**: Accepted
- **Related**: open-questions #2

## Context

「コード変更を Task 単位で持つ」(ADR候補: 検証経路と並ぶ中核)を実装する際のストレージ表現を決める必要があった。

選択肢:
- (a) Git再利用: 内部的に Git を使い、Task ごとに内部ブランチを自動生成。ユーザーには見せない
- (b) 独自ストレージ + Git互換出力: 変更を構造化して保持し、必要なら Git 形式に出力
- (c) 完全独自: 意味レベル差分を最初から持つ

## Decision

内部ストレージとして **Git を再利用**する。Task ごとの変更セットは内部ブランチ(または同等の参照)として表現され、ユーザー(人間/エージェント)には Git 概念を露出しない。

将来的に意味レイヤー(b相当)を上に被せる可能性は残す。完全独自(c)は採らない。

## Rationale

- Git の成熟したマージ・diff・ストレージを再利用できる
- 「Git はインフラに降格」(concept.md §4.1)と整合
- 実装コストが最小で、初期リリースの実現性が高い
- 意味レベル差分(c)は何十年挑戦されて成功していない領域(concept.md §4.3)であり、最初から狙うのはリスク大
- 段階的アプローチが可能: v1 Git再利用 → 必要なら意味レイヤーを被せる(LSP がファイルシステムの上に被さるように)

## Consequences

- Git は隠された実装詳細。CLI/MCP では「ブランチ」「コミット」を一級概念として露出しない
- 人間のエスケープハッチ(concept.md §4.4)としてGit層へ降りる手段は別途設計が必要(緊急時に内部の Git リポジトリに直接アクセスできる経路)
- マージ・コンフリクト検出はGit由来の制約(テキスト行単位)を引き継ぐ
- 意味レベルの調停が必要になった時点で意味レイヤーを追加で設計する
