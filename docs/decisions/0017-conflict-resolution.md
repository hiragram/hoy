# 0017. 統合時コンフリクトは自動リベース→失敗時にhookで差し戻し

- **Status**: Accepted
- **Related**: open-questions #2

## Context

ADR 0014 で Task 完了時の即時統合を採用した結果、別の Task が先に統合された後で本 Task が統合しようとして衝突するケースが発生する。この際の処理方針を決める必要があった。

選択肢:
- (a) 即座にエージェントへ差し戻し
- (b) daemon が自動リベースを試行
- (c) コンフリクト解決Taskを自動生成

## Decision

**(a) と (b) のハイブリッド**:

1. daemon がまず**自動リベース**を試みる
2. リベースがテキスト的にクリーン → 検証経路を**再走**させた上で、すべて passed なら統合
3. リベース失敗 or 検証経路の再走で fail → `task.rework_requested` イベントを発行し、hook 経由でclaim者に差し戻し

差し戻し時、Task の claim は保持される(他エージェントに奪われない)。

## Rationale

- 自動リベースで解決できるケースをエージェントに戻すのは無駄(待ち時間・推論コスト)
- 一方、テキスト的にコンフリクトしなくても意味的に壊れている可能性があるため、検証経路の再走は必須
- (c) は重く、初期スコープには大きすぎる
- 「ガードレール通過 → 即マージ」(ADR 0014)の建前を保ちつつ、衝突時のフォールバックを明確化

## Decision詳細

### 検証経路の再走

リベース後は passed 状態を一旦リセットし、必須(`required: true`)check を再走させる。テキストコンフリクトしないからといって安全とは限らないため。

### 差し戻し時のペイロード

`task.rework_requested` イベントには以下を含める(JSONペイロード):

- コンフリクトしたファイルと行範囲
- 相手側Task ID と Intent ID(参照したい場合に辿れる)
- リベース失敗箇所の詳細 or 再走で fail した check の情報
- 差し戻し理由コード

### claim保持

差し戻し中も claim は保持される。エージェントは作業を引き継いで再実装できる。明示的に release しない限り claim は継続する。

## Consequences

- daemon にはGitリベース相当の処理能力が必要(ADR 0013でGitを内部利用するため自然に得られる)
- 検証経路再走のコスト(特に automated check の実行時間)が統合のクリティカルパスに乗る
- 同時に多数のTaskが完了した場合、検証経路の再走が逐次化されてスループットに影響する余地あり → 将来の最適化課題
- セマンティック(意味レベル)のコンフリクト検出は本ADRでは扱わない。テキストコンフリクトしないが意味的に壊れるケースは検証経路の品質に依存
