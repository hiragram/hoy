# 0029. MVPはローカル単一開発者スコープ、マルチ開発者は将来拡張

- **Status**: Accepted
- **Related**: open-questions #5

## Context

ADR 0015 で nextgit はローカル常駐 daemon と決めたが、複数開発者・複数マシンでの協働モデルは未定だった。分散モデルとして以下の選択肢があった:

- (a) ローカル完結(マルチ開発者は当面スコープ外)
- (b) git的分散(各開発者が全状態を持ち push/pull で同期)
- (c) hub-and-spoke(中央originがsource of truth)
- (d) federation(P2P同期)

## Decision

**MVPは(a) ローカル単一開発者スコープ**。マルチ開発者対応は将来拡張として保留する。

将来拡張時の本命候補は **(c) hub-and-spoke**(claim の整合性が自然に解けるため)。ただし MVP 検証で得られた知見次第で再評価する。

## Rationale

- claim という排他リソースは分散モデル(b)(d)と根本的に相性が悪い
- 単一開発者スコープであれば claim 整合性は自明に解決し、設計・実装が大幅にシンプルになる
- 「単一開発者+複数エージェント」という構図は既に十分に新規性があり、MVPの差別化価値として成立する
- マルチ開発者対応を初期から設計に組み込むと、データモデル・プロトコルが過度に複雑化するリスク
- (c) を将来拡張で入れる場合、本MVPの daemon を「originの一形態」として再利用可能(ローカルdaemonがclaim裁定者であることは、hubと役割が同型)

## Consequences

- MVPでは複数マシン・複数開発者の同期は提供しない
- リポジトリの共有が必要なら git remote を使う(コードのみ。Intent/Task/claimは共有されない)
- 単一開発者向けに最適化された UX を最初に磨ける
- マルチ開発者が必要になった段階で、(b)(c)(d)の選択を MVP の使用実態を踏まえて行える
- 「会社で複数人が使いたい」というユースケースは初期スコープ外として明確化する
- open-questions #6(エコシステム移行戦略)はマルチ開発者前提の議論を含むため、本ADRと連動して更新が必要
