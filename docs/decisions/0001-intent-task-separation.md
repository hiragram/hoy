# 0001. IntentとTaskを別データ構造として分離

- **Status**: Accepted
- **Related**: open-questions #1

## Context

「意図 → タスク → 検証 → 統合」という中心概念のうち、Intent(意図)とTask(タスク)をデータモデル上どう持つかが未定義だった。混ぜて持つ案、片方を他方の属性として持つ案などが考えられた。

## Decision

IntentとTaskは別のデータ構造として保持する。すべてのTaskはいずれかのIntentに紐づく(Task → Intentの参照を持つ)。

## Rationale

- **責務が違う**: Intent は「なぜ・何を達成したいか」(変わりにくい、人間の領分が強い)。Task は「何をやるか」(可変、実装の都合で増減する)。
- **ライフサイクルが違う**: Intent は追記・改訂の履歴を厳格に管理したい。Task は気軽に作って消せる方がよい。混ぜるとどちらの作法に合わせるかが常に曖昧になる。

## Consequences

- Intent の改訂履歴と Task の増減履歴を別々に追える
- すべての Task が Intent に紐づくため、「なぜこの作業をしているのか」が常に辿れる
- Intent のないアドホックな Task は許容しない(必ず Intent を作る必要がある) ← 運用上のオーバーヘッドになる可能性は留意
