# エージェント時代の開発プラットフォーム コンセプト

> このドキュメントは hoy の思想・全体像を示すコンセプトドキュメント。
> 個別の設計判断は `docs/decisions/` 配下のADR、未決事項は `docs/open-questions.md` を参照。

## 1. 問題意識: GitHubは人間前提すぎる

GitHubは「人間がコードを書き、レビューし、マージする」ワークフローを前提に設計されている。AIエージェントが開発の主役になりつつある今、この前提と現実の間に様々な摩擦が生じている。

### 1.1 具体的な摩擦ポイント

**PRとコードレビュー**

- AIが1分で100ファイル変更するのに、PRという人間的な儀式が残っている
- 差分を「読んで理解する」というUX自体が人間向け

**レビュー指摘の管理が厳密でない**

- 「Resolved」ボタンは見た目を畳むだけで、実際の対応状況と紐付いていない
- 「どのコミットで対応したか」が構造化されていない
- 指摘ステータスが「未対応 / 対応済み(未検証) / レビュアー承認済み」の三段階になっていない
- 再レビュー時に「前回指摘からの差分」だけをフォーカスできるビューがない

**エージェント時代に特に深刻な影響**

- Claude Codeなどにレビュー対応を投げると、指摘の状態管理がGitHub側にないため、その責務が全部エージェントのコンテキストウィンドウに乗っかってしまう
- セッションが切れると「どこまでやったか」が消える
- どの指摘が未対応かをエージェント自身が正確に把握する手段がない

**その他の構造的問題**

- Issue・コメント文化: 非同期議論の積み重ねがエージェントにはノイズになる
- ブランチ戦略: 並列で数十タスクを走らせるときの粒度と合わない
- git blame: 「なぜ書いたか」がプロンプトの中にあってコミット履歴に残らない

## 2. 設計思想

### 2.1 中心概念の転換

```
今: コード → PR → レビューコメント → マージ
次: 意図(Intent) → タスク(Task) → 検証(Verification) → 統合(Integration)
```

「コード」と「人間のコミュニケーション」が中心から、**「意図」と「タスク」**が中心へ。

### 2.2 各コンポーネントの変化

**リポジトリ → プロジェクトグラフ**

- コードはあくまで成果物
- 「なぜこうなっているか」の意思決定ツリーがfirst-classに保存される

**PR → タスクの完了検証**

- タスクの完了条件(検証経路)が最初に定義される
- エージェントが対応したら条件に対して自動検証される
- 人間のレビューは「全差分を読む」から「検証を通過したものへの承認」に変わる

**コメント → 指摘チケット**

- レビュー指摘が最初からチケットとして生まれる
- ステータス・担当エージェント・対応コミットが構造化されている
- 条件を満たしたら自動クローズ

**ブランチ → エージェントの作業スコープ**

- Intent単位で1エージェントが排他的に作業権(claim)を持つ
- 統合時のコンフリクトは自動rebase + 失敗時の差し戻しで処理(ADR 0017)

### 2.3 人間の役割の再定義

- 意図・完了条件を定義する
- エージェントが判断できない設計上のトレードオフを決める
- 検証ロジック自体が正しいかをレビューする

コードを読むのではなく、**「何を作るか」と「正しく作れたかの基準」を管理する役割**になる。

## 3. コアデータモデル

hoy のすべては以下の概念の上に立っている。詳細は対応するADRを参照。

### 3.1 Intent(意図)

「なぜ・何を達成したいか」を表す一級概念。

- ID は安定、内容は version で更新される(ADR 0008)
- 入れ子可能(親子関係を持てる)(ADR 0004)
- ステータスは `active / closed`、closing reason 付き(ADR 0019)
- closed Intent は reopen 不可(ADR 0023)
- 1 Intent は1 Principal による排他 claim 下に置かれる(ADR 0009)
- 親と子は独立して claim 可能(ADR 0010)

### 3.2 Task(タスク)

「何をやるか」を表す実装の自然な単位。

- 必ずいずれかの Intent に紐づく(ADR 0001)
- 入れ子不可、フラット(ADR 0004)
- 起票主体は人間/エージェント両方を許容、`created_by` メタデータで識別(ADR 0003 / 0006)
- Intent@version への依存を `depends_on` で表現できる(ADR 0018)
- 完了時に即時統合される(ADR 0014)
- `completed → reverted` の遷移を一級操作として持つ(ADR 0034)

### 3.3 Verification Path(検証経路)

「タスクの完了をどう確認するか」を構造化したもの。詳細は §5。

### 3.4 Claim(作業権)

並列エージェントの調停の基盤。

- Principal 単位で確立される(ADR 0025)
- ハートビート + 強制release で生存管理(ADR 0011)
- 書き込み排他、読み取り自由(ADR 0012)

### 3.5 監査ログ

全状態変更操作を append-only な統一ストリームとして記録(ADR 0027)。

## 4. アーキテクチャ

### 4.1 ローカル常駐 daemon

hoy はローカルマシン上で動く daemon として実装される(ADR 0015)。GitHub に対する git のように、リモートサーバは初期スコープ外。

daemon の責務:

- 状態管理(Intent / Task / 検証経路 / claim)
- 統合処理とコンフリクト検出
- claim 管理(ハートビート、強制release)
- MCP サーバとして agent からの接続を受ける
- イベント発行と hook 起動

### 4.2 エージェント連携

- hoy 自身はエージェント機能を持たない(ADR 0002)
- エージェントは MCP 経由で daemon に接続
- daemon → エージェントへの通知は `agent-dispatch.sh` hook で行う(ADR 0016)
- hook はリポジトリに含まれる。エージェントの実起動はプロジェクト側の責務

### 4.3 CLI と MCP の対称性

- API(プロトコル)が一次
- CLI と MCP はどちらも API の薄いラッパー(ADR 0007)
- 「CLI にあるが MCP にない操作」のような非対称は許さない

### 4.4 認証

- Principal 単位の token-based 認証(ADR 0025)
- 1 Principal が複数 Session を持てる(hook で起動した子プロセスも同じ Principal として動ける)
- 細粒度な capability は将来拡張(ADR 0026)

### 4.5 機密管理

- daemon は機密管理機構を内蔵しない
- OS のファイル権限 + 外部 secret manager 前提(ADR 0028)

## 5. 検証経路(Verification Path)

「タスクの完了をどう確認するか」はこのプラットフォームの中核。自動検証だけで完結しないタスクが現実には大量にあるため、検証経路そのものをデータ構造として持つ。

### 5.1 基本的な考え方

エージェントコーディング時代における前提:

- **ガードレールとしての自動テストは必須**
- **とはいえ人間が動作確認しないといけない、自動テストでは担保しきれない領域がある**
- **あるタスクや変更内容がどちらに当たるのかは明示されるべき**

### 5.2 これがプロトコルに乗ると効く理由

**検証経路がメタデータになる**

タスクに検証経路フィールドが乗ると、エージェントが「自分で完了宣言していいか」を判断できる。今はここがエージェントの空気読みに任されていて事故る。

**人間のレビュー帯域を正しい場所に集中できる**

`human-required`が明示されていれば、人間は「自動検証が通ったやつをスタンプ承認」じゃなく「そもそも自動では担保できない部分」に時間を使える。逆に`automated`タスクの差分を人間が読みに行くのは無駄だと宣言できる。

**「どっちに当たるか」自体がレビュー対象になる**

エージェントが「これは自動テストで足りる」と判断したのが妥当か、を人間が承認する層ができる。判断ミスがあれば`automated → human-required`に格上げするフローがプロトコルに乗る。これは今のCIには無い概念。

### 5.3 1タスクは複数の検証経路を持つ

検証経路は単一の値ではなく、複数の経路をリストとして持つ。

例:

```
[automated(unittest), automated(vrt), human(ux), human(performance)]
```

#### なぜサブタスク分割ではなく、1タスクに複数経路を持たせるのか

「APIエンドポイント追加」のようなタスクの中に、ロジック部分(automated可)とエラーレスポンスのUX(human必要)が混在することは普通にある。これをサブタスクに割って各々に経路を持たせる案も考えられるが、以下の理由で採らない。

1. **分割の単位がコードの単位と一致しない**: 実装は同じハンドラ関数の中に混ざって書かれる。サブタスクごとにコミットやブランチを分けようとすると、コード上は不可分なのに管理上だけ分かれるという歪みが出る。
2. **分割粒度が事前に確定しない**: 実装してみて初めて「ここは人間確認が要る部分だ」と気づく。事前にサブタスク分割を強制すると、実装中に分割をやり直すコストが常に発生する。
3. **依存関係が増えて管理が爆発する**: サブタスクA(automated)とサブタスクB(human-required)が同じ関数を触っていたら、Aだけマージみたいなことはできない。結局束で扱うことになる。

逆に、1タスクに複数の経路を持たせる方式なら:

- 実装の自然な単位(=タスク)を壊さない
- 実装過程で「ここは人間確認要るな」と気づいたら経路を追加すれば済む
- 完了条件は「全(必須)経路が通ること」というAND条件で素直に表現できる

**分類は実装より後から確定する**ので、事前分割を強制する設計は実装フローと噛み合わない。

### 5.4 データ構造

```
VerificationCheck {
  id: string              // 安定ID
  kind: "automated" | "human"
  category: string        // unittest, vrt, ux, performance, security, ...
  spec: ...               // kindごとに中身が違う
  status: pending | running | passed | failed | waived
  evidence: ...           // 通った根拠(ログ/スクショ/承認者)
  required: bool          // 完了に必須か、参考情報か
}
```

設計上のポイント:

1. **`kind`と`category`を分ける**: 二段にしておくと、「automatedな検証だけ全部走らせる」「カテゴリがperformanceのものだけ抽出」両方のクエリが効く
2. **`spec`の中身は`kind`で型が変わる**:
   - automated: 実行コマンド、成功条件
   - human: 誰が・何を確認するか・どの環境で
3. **`evidence`が一級市民**: 「なぜマージできたか」を後から辿れる。git blameの正常進化系
4. **`waived`状態を入れる**: 「このチェックは今回スキップする(理由付き)」を正規化。ただし誰が・なぜwaiveしたかは必ず残す
5. **`required`フラグ**: 全部が完了に必須とは限らない

検証間の依存関係はリストの並び順ではなく**別フィールド**で持つ。

### 5.5 「どっちに当たるか」の判定責任

検証経路の選択をエージェントだけに任せると、エージェントは自分に都合よく`automated`に倒しがち。

→ プロジェクトレベルで「このディレクトリ配下の変更は必ずhuman-required」のように**ポリシーをコードベース側に宣言**できるようにする。CODEOWNERSの進化系のような位置付け。

### 5.6 統合時の検証経路再走

ADR 0017 により、統合時にコンフリクトが起きた場合は自動 rebase 後に検証経路を再走させる。テキストコンフリクトしないからといって意味的に安全とは限らないため、必須 check を再度走らせる。

## 6. gitはインフラに降格する

### 6.1 daemonの内部実装としてのGit

hoy は変更セットの保管に内部的に Git を使う(ADR 0013)。ただし「ブランチ」「コミット」をユーザーには露出しない。Git は隠された実装詳細。

```
今: 人間 → git コマンド → GitHub
次: 人間 → エージェント → MCP → hoy daemon → Git(内部)
```

TCPのように、存在はするが普通の人は意識しない層になる。

### 6.2 置き換えられる部分

- **ブランチとマージ**: 並列エージェントが claim を取って働く世界では、人間が命名・管理するモデルが合わない
- **コミットという単位**: 意味のある単位は「Task の完了」(ADR 0014)
- **コンフリクト解決**: テキストエディタで目視で直す UX は消え、自動 rebase + 検証経路再走 + 失敗時 hook で agent に差し戻し(ADR 0017)

### 6.3 意味レベル差分について

gitの根本的な限界は「**テキストの差分しか知らない**」こと。次世代は意味レベル差分を持てるかもしれない。

ただし「意味レベルの差分」は Semantic merge、AST-based VCS、Unison、Pijul など何十年も挑戦されて成功していない領域。テキストdiffが残ってる理由は怠慢ではなく「言語非依存・ツール非依存・人間が読める」という三点が強すぎるから。正面から倒すよりは、**テキストdiffの上に意味レイヤーを被せる**方が現実解。LSPがファイルシステムの上に被さっているのと同じ構図。

ADR 0013 でも当面はテキストレベル差分のまま。意味レイヤーは将来拡張として残す。

### 6.4 人間のエスケープハッチ

「人間はgitを直接触らない」を厳格に運用すると、エージェントがやらかしたときに降りていって直す層が消えてしまう。

- 通常運用では Git は隠された実装詳細
- 緊急時は Git に直接降りられる経路を残す(ADR 0034 で reconciliation コマンドが必要と明記)
- 暴走agent検知や自動制止はhoy内蔵せず、外部 hook で実装(ADR 0035)

## 7. 設計の順番

GitHubの失敗パターン: 人間向けのUIとワークフローを先に作って、後からAPIやエージェント対応を継ぎ足した。

### 7.1 逆順にする

**① エージェントファーストなプロトコルとデータモデル**

- 「意図」「タスク」「検証経路」「claim」を first-class で持つ
- API が一次、CLI と MCP は薄いラッパー(ADR 0007)
- フル OSS(ADR 0030)

**② daemon としての実装**

- ローカル常駐(ADR 0015)
- 単一開発者スコープから始める(ADR 0029)
- データモデルは MVP で完備、UI・運用ツールは後(ADR 0031)
- 外部ツール統合(GitHub Issues / Linear / IDEプラグイン等)は MVP では持たない(ADR 0032)

**③ ホスティング・マルチ開発者対応**

- マルチ開発者対応の本命候補は hub-and-spoke(claim整合性が綺麗に解ける)
- MVP 検証で得られた知見次第で再評価

### 7.2 gitの設計哲学に学ぶ

Linusがgitを作ったとき、UIより**データモデルを先に正しく設計した**。「内容アドレスストレージ」という核があって、その上にコマンドが乗った。

同じように、**エージェント時代のバージョン管理のデータモデルを正しく設計することが先**で、UIはその後でいい。

### 7.3 段階的アプローチ

- v1 (MVP): 単一開発者・ローカル daemon・データモデル完備
- v2: マルチ開発者対応(hub-and-spoke候補)、運用ツール充実
- v3+: 意味レベル差分レイヤー、リッチなUI、エコシステム統合

## 8. まとめ

| 観点 | 現在(GitHub) | hoy |
|------|-------------|--------|
| 中心概念 | コード・人間のコミュニケーション | Intent / Task / Verification / Claim |
| レビュー | 差分を人間が読む | 完了条件への自動検証 + 必要に応じて人間承認 |
| 指摘管理 | コメント(非構造化) | チケット(構造化・自動クローズ) |
| 検証 | CI(automated)のみ構造化 | automated/humanを並列に構造化、required/waived付き |
| ブランチ | 人間が命名・管理 | Intent claim(自動・排他・Principal単位) |
| Git | 人間が直接操作 | daemon の内部実装(隠蔽、緊急時のエスケープあり) |
| 統合 | PRレビュー後に手動マージ | 検証パス後に即時統合、コンフリクトは自動rebase + 差し戻し |
| 取り消し | git revert | Task `reverted` 状態への遷移(裏で git revert) |
| 人間の役割 | コードを書いてレビューする | 意図と基準を定義して承認する |
| 設計順序 | UI → API | データモデル → プロトコル → daemon → UI |
| 範囲 | チーム協業ツール | (MVP)単一開発者+複数エージェント |
